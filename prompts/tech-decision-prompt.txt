- The "Tech stack TBD" notes scattered across documents

- Timeline constraints (MVP in 10-14 weeks)

CRITICAL CONTEXT — MULTI-PROJECT ARCHITECTURE:

This is NOT a single mobile app. The system consists of multiple applications 
that share a backend:

1. **Mobile App** (Android + iOS) — The consumer-facing driver app. This is 
   the MVP's primary interface. Car enthusiasts use this daily.

2. **Web Application** — For MVP, this is admin-only (the founder uses it to 
   manage users, view analytics, moderate content, manage challenges). In future 
   phases, this becomes user-facing — drivers can optionally use the web version,
   and B2B users (garages, dealers, fleet managers) will primarily use the web app.

3. **Backend API** — Shared API that serves both mobile and web. Must be designed 
   as a single source of truth that multiple clients consume.

4. **Database** — Single database shared across all applications.

This multi-project architecture affects every tech decision:
- The API must be client-agnostic (mobile and web consume the same endpoints)
- Auth must work across mobile and web
- The web framework choice is a separate decision from the mobile framework
- Code sharing between projects (shared models, validation logic) is desirable
- The solo developer must maintain all projects — simplicity and consistency matter

Present this summary to the user before proceeding. Ask them to confirm or 
correct the multi-project scope.

---

STEP 2: For each technology decision below, do the following:

a) State what the project needs based on the documents (not assumptions)
b) Present 2-4 options with pros and cons evaluated against:
   - Developer's existing skills (ASK the user about their experience)
   - Cost at small scale (0-1,000 users)
   - Cost at medium scale (1,000-10,000 users) 
   - Solo developer productivity (speed of development)
   - Cross-platform quality (Android + iOS)
   - Scalability path (from MVP to Phase 3)
   - Community and ecosystem (documentation, packages, support)
   - Claude Code compatibility (how well can AI assist with this tech)
c) Give your recommendation with clear reasoning
d) Ask the user any clarifying questions needed
e) Wait for the user's decision before moving to the next area

THE 12 TECHNOLOGY DECISIONS:

### Decision 1: Mobile Framework
Options to evaluate:
- .NET MAUI (C#, XAML)
- Flutter (Dart)
- React Native (JavaScript/TypeScript)
- Kotlin Multiplatform (Kotlin)

Questions to ask the user:
- What languages are you most experienced with? (C#, Dart, JavaScript, Kotlin, other)
- Have you built a mobile app before? With which framework?
- Do you have strong preferences for any language or ecosystem?

### Decision 2: Web Framework
Options to evaluate:
- Blazor (C# — shares code with .NET backend and potentially MAUI)
- Next.js (React — large ecosystem, good for admin + future user-facing)
- Angular (TypeScript — enterprise-grade, good for complex admin panels)
- Vue.js / Nuxt (lightweight, easy to learn)
- Razor Pages / MVC (if staying full .NET)

Consider: For MVP, this is admin-only. But it WILL become user-facing and 
B2B-facing in Phase 2-3. Choose something that scales from "simple admin panel" 
to "full B2B application for garages, dealers, fleet managers."

Questions to ask the user:
- Do you have frontend web experience? Which frameworks?
- How important is code sharing between mobile and web?
- Would you prefer the same language across mobile, web, and backend?

### Decision 3: Backend Approach
Options to evaluate:
- Custom API with ASP.NET Core
- Custom API with Node.js/Express or Fastify
- Backend-as-a-Service: Supabase (Postgres + Auth + Storage + Realtime)
- Backend-as-a-Service: Firebase (Firestore + Auth + Storage + Functions)
- Hybrid: BaaS for auth/storage + custom API for business logic

IMPORTANT: The backend must serve BOTH the mobile app AND the web application 
through the same API. Design it as client-agnostic from day one.

Questions to ask the user:
- How comfortable are you writing and maintaining a backend API?
- Do you prefer full control over the backend or faster time-to-market with BaaS?
- Are you comfortable with serverless/cloud functions?

### Decision 4: Database
Options to evaluate:
- PostgreSQL (self-hosted or managed)
- SQL Server (if staying in .NET ecosystem)
- Supabase Postgres (managed, includes API layer)
- Firebase Firestore (NoSQL, document-based)
- PlanetScale / Neon (serverless Postgres)

Consider: The data model includes users, organizations, vehicles, expenses, 
maintenance records, reminders, challenges, leaderboards. This is relational data.
Multi-role architecture requires proper data isolation.
Single database shared by mobile app, web app, and backend.

### Decision 5: Authentication
Options to evaluate:
- ASP.NET Identity (if using .NET backend)
- Firebase Auth
- Supabase Auth
- Auth0
- Clerk

Requirements from documents:
- Email + password registration
- Must support multiple user roles from day one (driver now, garage_owner/dealer/fleet_manager later)
- Social login (Google, Apple) — nice to have for MVP
- Session management for BOTH mobile app AND web app
- Single auth system across all applications

### Decision 6: Hosting & Deployment
Options to evaluate:
- Azure (App Service + managed DB)
- AWS (ECS/Lambda + RDS)
- Railway
- Fly.io
- Render
- Supabase (if using BaaS)
- Vercel + Supabase (if using Next.js API routes)

Consider: Need to host backend API + web application + database.
Mobile app is deployed via App Stores (separate process).

Questions to ask the user:
- Do you have existing cloud accounts or credits?
- Budget for infrastructure per month? (The monetization plan estimates €50-200/month post-launch)

### Decision 7: Image Storage
Options to evaluate:
- Azure Blob Storage
- AWS S3
- Cloudflare R2 (S3-compatible, no egress fees)
- Supabase Storage
- Firebase Storage

Requirements: Vehicle timeline photos, compressed to max 1MB each. 
At 5,000 users with avg 20 photos = ~100K images = ~100GB.
Images must be accessible from both mobile and web apps.

### Decision 8: Push Notifications
Options to evaluate:
- Firebase Cloud Messaging (FCM) + Apple Push Notification Service (APNs) directly
- OneSignal (wrapper, free tier available)
- Azure Notification Hubs
- Expo Push (if using React Native)

Requirements: Maintenance reminders, challenge updates. 
Not real-time critical — scheduled notifications.
Mobile only (web notifications can come later).

### Decision 9: Analytics & Telemetry
Options to evaluate:
- PostHog (open source, self-hostable)
- Mixpanel
- Firebase Analytics
- Amplitude
- Custom events to own database

Requirements from documents:
- Must track journey success metrics (activation, retention, feature usage)
- Must track paywall trigger conversion rates
- Must track which features drive premium conversion
- Track across BOTH mobile and web
- GDPR compliant

### Decision 10: In-App Purchases & Subscription Management
Options to evaluate:
- RevenueCat (wraps Apple/Google billing, free tier up to $2.5K MRR)
- Direct Apple StoreKit + Google Play Billing
- Adapty
- Qonversion

Requirements: €2.99/month and €24.99/year premium subscription.
Apple and Google take 30% (15% for small business program).
Web-based subscription purchase may be considered in future (to avoid store fees).

### Decision 11: CI/CD Pipeline
Options to evaluate:
- GitHub Actions
- Azure DevOps Pipelines
- Bitrise (mobile-focused)
- Codemagic (Flutter/mobile focused)
- Fastlane (mobile deployment automation)

Consider: Need CI/CD for mobile builds (Android + iOS), web deployment, 
and backend deployment. Ideally one pipeline system for all projects.

Questions:
- Are you using GitHub, GitLab, or Azure DevOps for source control?

### Decision 12: Error Monitoring
Options to evaluate:
- Sentry (free tier: 5K events/month)
- Firebase Crashlytics (free)
- AppCenter (Microsoft, being retired — avoid)
- Bugsnag

Requirements: Solo developer needs automatic crash reporting.
Can't rely on users reporting bugs.
Need error tracking on mobile AND web AND backend.
